## 機能要件

### FR-01: 受信郵便物の登録・管理

- 受信した郵便物を登録できる
- 登録項目：受取日、送り主（名称・住所）、宛先担当者、郵便物種別（手紙・書類等）、メモ
- 登録済み郵便物の一覧表示
- 郵便物の編集・削除（管理者のみ）

### FR-02: 業務フロー（受信ワークフロー）

郵便物が届いた際の処理フロー：

```
受信郵便物登録 → 担当者へ Slack/Teams 通知 → 担当者が受取確認 → ステータス「完了」に更新
```

- ステータス管理：`未通知` / `通知済み` / `受取確認済み（完了）`
- 担当者が受取確認を行うと、郵便物のステータスが「完了」に変わる

### FR-03: ユーザーロール管理

| ロール | 権限 |
|---|---|
| **管理者** | ユーザー管理、全郵便物の登録・閲覧・編集・削除、ダッシュボード閲覧 |
| **担当者** | 自分宛て郵便物の受取確認、全郵便物の一覧閲覧（閲覧のみ） |
| **閲覧者** | 郵便物一覧の閲覧のみ |

### FR-04: チャット通知連携（Slack / Teams）

- 郵便物登録時に Slack または Teams の Webhook 経由で通知を送信
- 通知先：宛先担当者（または指定チャンネル）
- 通知内容：受取日、送り主、郵便物種別

### FR-05: 検索機能

- 基本検索：日付範囲、送り主、宛先担当者による絞り込み
- ステータスによるフィルタリング（未完了 / 完了）

### FR-06: 統計ダッシュボード

- 月次統計：月別受信郵便物件数のグラフ表示
- 年次統計：年別受信郵便物件数の集計
- 担当者別・種別別の受信件数サマリ

### FR-07: データ保管・削除

- データ保管期間：3ヶ月〜1年
- 保管期間を超えたデータの自動削除またはアーカイブ機能（管理者操作）

## 非機能要件

### NFR-01: 認証・認可

- **認証方式**: シンプルな ID/パスワード独自認証（学習目的）
- **認可**: ロールベースアクセス制御（RBAC）—FR-03 のロール定義に基づく
- セッション管理（詳細は NFR Requirements ステージで決定）

### NFR-02: セキュリティ

- セキュリティ拡張ルール: **無効**（PoC・学習目的のためスキップ）
- 基本的な入力バリデーション・XSS 対策は実施

### NFR-03: スケーラビリティ・パフォーマンス

- 対象: 小規模チーム（数十名規模）
- 拡張性要件: なし（現行要件の充足のみ）
- 詳細な性能要件は NFR Requirements ステージで定義

### NFR-04: テスト戦略

- **Property-Based Testing (PBT)**: 全ルール（PBT-01〜PBT-10）を強制適用
  - 純粋関数・シリアライズ処理・ビジネスロジック全域に PBT を適用
  - PBT フレームワークは技術スタック決定後に選定（PBT-09 準拠）
  - 例ベーステストと PBT の併用が必須（PBT-10 準拠）

### NFR-05: 技術スタック

- アーキテクチャ: フルスタック Web アプリ（フロントエンド ＋ バックエンド API）
- 詳細な技術スタック（フレームワーク・言語・DB 等）は NFR Requirements ステージで決定

### NFR-06: データ管理

- データ保管期間: 3ヶ月〜1年（FR-07 参照）
- 削除ポリシー: 管理者が手動でアーカイブまたは削除を実行

## ユーザーシナリオ（主要ケース）

### シナリオ 1: 郵便物受取登録（管理者）

1. 管理者がログイン
2. 受信郵便物を登録（送り主・担当者・種別を入力）
3. 登録完了後、宛先担当者に Slack/Teams 通知が自動送信される

### シナリオ 2: 受取確認（担当者）

1. Slack/Teams 通知を受けた担当者がシステムにログイン
2. 自分宛ての郵便物一覧を確認
3. 対象郵便物の「受取確認」ボタンをクリックし、ステータスを「完了」に更新

### シナリオ 3: 統計確認（管理者・閲覧者）

1. ダッシュボードにアクセス
2. 当月・当年の受信件数グラフを確認
3. 担当者別件数サマリを参照

- システムは学習目的のため、シンプルな実装を優先する
